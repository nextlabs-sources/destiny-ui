The Quick Start Guide explains how to use sample ABAC (Attribute Based Access Control) policies and configure a sample application to make authorization requests in NextLabs Cloud Authorization Service. This service is referred to as CloudAz.

After completing the Quick Start Guide, you will:

•Understand how a simple authorization policy is used in CloudAz

•Be able to perform a simple authorization using CloudAz

Before you begin

To use the Quick Start Guide example, you need to import sample ABAC policies into your CloudAz instance. The sample policies are for authorization in a typical Support Ticket system. There are three authorization rules to be enforced by the Support Ticket system:

•Allow support representatives to edit and reassign tickets that belong to their product area

•Allow users from the IT department to view all support tickets

•Deny access to security vulnerabilities if they are not created by or assigned to the user

import sample ABAC policies

Run the Java sample application

After you have imported the sample policies, you can run the Java application, which sends a set of authorization requests to the CloudAz REST API. CloudAz evaluates each authorization request against the imported sample policies; calculates a decision, allow, deny, or indeterminate; and returns the decision with an optional set of obligations. Obligations are additional instructions the sample application needs to perform before proceeding with the application logic. The sample application displays each set of request attributes and the response on-screen before proceeding to the next request.

Procedure

1Set up the Java SDK with JDK 7 or higher. For instructions, go to http://docs.oracle.com/javase/7/docs/webnotes/install/.

2Download the sample Java application from {{splashScreen.linksSaas.sampleCode}}

3Extract the files from the downloaded package.

4In the extracted folder, open the java folder, which has the required binaries and samples for the Java API.

Note: Files are extracted with the Read-only attribute set in their properties. To make the files editable in Windows, right-click the file(s), then click Properties, make sure the read-only checkbox is not checked.

5Open sample_code/config/openaz-pep.properties file in a text editor. Replace the following placeholders with the properties for your system and save the file:

•<CloudAz REST API host>: The hostname of your CloudAz instance, which was provided in the Welcome email from NextLabs- {{jpcDomain}}

•<LOGIN_USER_NAME>: The username of your account. For example, Administrator

•<LOGIN_PASSWORD>: The password associated with your account.

6Send the sample authorization request

aOpen a Windows Command prompt or Linux terminal, and navigate to sample_code folder.

bRun one of the following scripts to send the request:

- Windows: cloudaz_request.bat

- Linux: cloudaz_request.sh

The response appears in the console. To learn more about the response, refer to About the sample client section below.

About the sample client:

The sample client performs the following tasks:

1Prepares each authorization request by collecting all required attributes. These include user, resource, and environment attributes, as well as the action that the user is performing, which are considered by the CloudAz policy engine, the Policy Decision Point (PDP).

2Submits the REST authorization request to the CloudAz PDP over HTTPS with appropriate credentials.

3Processes the response from the PDP and handles any obligations. Typically, the authorization decision is enforced in the application being securing.

How the sample requests are processed by the PDP

The PDP (Policy Decision Point) processes the sample authorization requests from both the Java and JavaScript sample applications as follows:

•AllowITDeptUsersToViewTickets: Chris Webber, a user who belongs to the IT department, is requesting access to view a ticket that belongs to the SharePoint product. The CloudAz PDP evaluates this request and responds with Allow, since the request matches the following policy: Allow users from the IT department to view all support tickets.

•AllowUsersToEditTktsAssignedToThem: Kate Adams, a user who belongs to the IT department, has these attributes: Her job role is support representative, and her product area is Exchange email. She is requesting access to edit support tickets in product Exchange email. The CloudAz PDP evaluates this request and responds with Allow, since the request matches the following policy: Allow support representatives to edit and reassign tickets that belong to their product area.

•DenyUsersAccessToSecurityTkts: Kate Adams, a user who belongs to the IT department, is requesting access to view a security category support ticket. The CloudAz PDP evaluates this request and responds with Deny, since the request matches the following policy: Deny access to security vulnerabilities if not created by or assigned to the user. Access is denied because although Kate belongs to the IT department, the support ticket is in the Security category and it is not assigned to her. The response also contains the obligation that the PEP (Policy Enforcement Point) needs to handle.

•NoMatchingPolicyFound: Kevin Chen, a user who belongs to the Marketing department, is requesting access to view a support ticket. The CloudAz PDP evaluates the request and responds with Indeterminate, since there is no policy that matches the attributes of either the user or the ticket that is being accessed. In such cases, the PEP determines how to handle the authorization request. As a good practice, the PEP should default to Deny when requests result in a response of Indeterminate.

For more information about CloudAz components such as the PDP and PEP, refer to CloudAz overview link.

Take it for a spin

•In NoMatchingPolicyFound, change the user’s department attribute to IT

•In DenyUserAccessToSecurityTks, change the resource attribute of the support ticket, and update the created_by attribute to kate.adams

Send an authorization request to view the new policy effect.
